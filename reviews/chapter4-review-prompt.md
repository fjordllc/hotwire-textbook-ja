# 重点的に見てほしい観点
1. 技術的正確さ（最優先）。誤解を招く記述がないか厳しく検証する。特に:
   - 「Rails 7+ では turbo-rails が標準で入るため、通常CRUDの時点で既に Turbo Drive が効いている」という主張
   - Rails 8 標準の認証ジェネレータの説明（User は email_address/password_digest のみで name を持たない、Session モデルが生成される、本書では name を後から追加する）
   - Task#status を todo/in_progress/done の enum にする設計の妥当性
   - 6モデル（User/Project/Task/Comment/Tag/Tagging）と関連（Project1-*Task、Task1-*Comment、Task*-*Tag、Task*-1 User）の正しさ
   - importmap を主軸にし、jsbundling と第22章/第6章へ送る方針の妥当性
   - 「すべてサーバーが HTML を返す」という操作ストーリーの記述が Hotwire の実態と一致するか
2. 学習設計。第4章が「仕様の共有」章として、初級者に過不足ない情報量か。先に全体像を渡す構成が機能しているか。後続章で参照される前提（status 値、UI を後回しにするモデル）が明確に固定されているか。
3. サンプルアプリ設計。スコープの線引き（認証=最小、認可=第31章、マルチテナントなし、タグ/コメントUIは後回し、通知は非永続）が初級者の学習に対して妥当か。Relay が各部の例を無理なく供給できるか。
4. 本文の質。STYLEGUIDE 準拠（です/ます調・短文・用語統一・「従来のRails→なぜ問題→Hotwireの解決」順）。分かりやすさ、冗長さ、表と本文のバランス。
5. 一貫性。OUTLINE / SUMMARY / 章番号参照（例: 第5章 5.10、第22章、第31章）との食い違い。用語の揺れ。
