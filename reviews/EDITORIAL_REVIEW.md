## 第1部: レビュー用プロンプト

```text

# 対象
- manuscript/ 配下の全原稿: preface.md、part1〜part10 の index.md / chapter*.md / exercises.md、appendix/*.md、afterword.md

# 見るべき観点
1. 構成と物語の弧: 「HTML over the wire」という縦軸と、各部の軸（visit / frame / 命令 / 振る舞い…）が、章をまたいで一貫して流れているか。読者が迷子にならない順序か。
2. 一貫性: 用語（STYLEGUIDE 準拠）、サンプルアプリ名（Relay）、コード規約（dom_id、422、partial 共通化）が全編で揃っているか。表記ゆれ・名称ゆれがないか。
3. 学習設計: 各章のねらいと結論が対応しているか。前方参照（「第N章で扱う」）が、その時点で読者に必要な情報を後回しにしていないか。難易度の段差が急すぎないか。

7. 前付け・後付け: まえがき・おわり・付録が、本編と矛盾なく、案内として機能しているか。
8. アクセシビリティと網羅: a11y・テスト・アンチパターンが各機能章に織り込まれ、抜けがないか。
```

## 第2部: 実施所見（全原稿を通読した結果）

### 強み

- <strong>テンプレートの徹底</strong>。全38章が「## この章のねらい → 節 → ## 参考資料 → 章末の接続引用（>）」で統一。読者がリズムをつかみやすい。
- <strong>縦軸と各部の軸</strong>。「HTML over the wire」を全編で繰り返しつつ、各部冒頭に軸（第3部=「すべては visit」、第4部=「frame は小さな visit 領域」、第5部=「命令を送る」、第6部=「状態を HTML に置く」…）を立て、章がその軸に従属している。物語の弧が明快。
- <strong>1 アプリを育てる一貫性</strong>。Relay を第2部の素の CRUD から段階的に Hotwire 化。前方・後方参照（dom_id=第17章、422=第8章、partial 共通化=第12/17章）が整合し、章をまたいで知識が積み上がる。
- <strong>表記・規約の遵守</strong>。禁止表記（コントローラー／バリュー等）ゼロ、強調は `<strong>`、422 は本文「422」＋コード `:unprocessable_entity`、章参照は範囲内。STYLEGUIDE 準拠が全編で保たれている。
- <strong>実務性</strong>。各機能章にアンチパターン・テスト・a11y が織り込まれ、第8部（保守）と第10部（選定・撤退判断）まで含む。「できる」でなく「保守できる」という編集方針が一貫。

### 改善点

#### Should-fix

1. <strong>「a11y」の初出に展開定義がない</strong>（初出: `part5/chapter17.md:127`、以降 Part 7 で節見出しに多用）。略語が定義なしで節タイトルに使われている。初出箇所で「アクセシビリティ（a11y）」と一度だけ展開し、以降は「a11y」とする旨を添えるべき。初級読者の最初の引っかかりを防ぐ。

2. <strong>まえがきがサンプルアプリを「Relay」と名指ししていない</strong>（`preface.md` は「1 つの Rails サンプルアプリ」と総称）。第1章以降は一貫して「Relay（仮称）」。まえがきでも名前を出し、「本書では Relay というタスク管理アプリを育てます」と予告すると、前付けと本編の連続性が増す。

3. <strong>図版が未制作で、仕組み章の密度が文章に集中している</strong>（`manuscript/figures/` は空）。本書は「サーバーとブラウザの間で何が流れるか」「frame の id 一致」「stream の DOM 操作」「broadcast の配信」など、図で一目で伝わる題材が多い。別添の図版計画（FIGURES.md）に沿って図を入れると、とくに Part 1（薄め）と長いコード章（厚め）の体験差が均される。Should-fix の理由は、文章だけでは初学者の理解負荷が高い箇所が複数あるため。

#### Nice-to-have

4. <strong>章末「接続引用（>）」が全38章で同型</strong>。「本章では…を学びました。次の第N章では…」という定型が続き、やや単調。要所（各部の最終章など）で、問いかけや要約の形に変化を付けると、通読時の単調さが和らぐ。

5. <strong>分量バランス</strong>。Part 1（第1〜3章、各 50〜60 行）が、実装章（第5章 288 行、第16章 214 行など）に比べ薄い。Part 1 は概念導入なので意図的だが、図版（fig-1-x, fig-2-x, fig-3-x）で視覚的な密度を補うと、序盤の手応えが増す。

6. <strong>長いコード章のコード前文脈</strong>。第16・24・26章はコードブロックが多い。おおむね各ブロック前に「何を変えるか」が置かれているが、第24章の `infinite_scroll_controller`（ターゲットコールバックの差し替え）など、図（fig-24-1）があると一段わかりやすい。

7. <strong>付録 G の「完成版コード解説」と実コードの対応</strong>。付録Gはファイル×役割×章の対応表として有用だが、実リポジトリ（完成版 Relay）への参照リンクがない。刊行形態に応じて、完成版コードの公開場所（GitHub 等）への導線を 1 行加えると、読者が手元のコードと照合しやすい。

### 刊行前に対応すべき上位 5 件

1. 「a11y」を初出（第17章）で「アクセシビリティ（a11y）」と展開定義する。
2. まえがきで、サンプルアプリ名「Relay」を名指しして予告する。
3. 図版計画（FIGURES.md）に沿って、最低でも各部の中核図（visit 置換、id 一致、stream の DOM 操作、broadcast、Stimulus ライフサイクル、Native の WebView+shell）を制作・配置する。
4. 章末接続引用の定型を、各部最終章など要所で少し崩し、単調さを減らす。
5. 付録Gに、完成版 Relay コードの公開場所への導線を追加する（刊行形態が決まり次第）。

> これらはいずれも本としての仕上げに関わるもので、技術的な内容の正しさ（各章のテクニカルレビュー済み）とは別レイヤーです。1〜3 を優先すると、初学者の通読体験が大きく改善します。
